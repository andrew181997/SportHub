## Описание проекта

SportHub — это веб-приложение, которое предоставляет организаторам любительских спортивных турниров инструменты для:

- Создания сайта лиги/турнира с настраиваемым оформлением
- Управления командами, игроками, тренерами и судьями
- Ведения календаря матчей с результатами в реальном времени
- Автоматического расчёта турнирных таблиц и статистики
- Публикации новостей, фото- и видеоматериалов

Целевая аудитория — организаторы любительских лиг по хоккею, футболу, баскетболу, волейболу и другим командным видам спорта.

## Функциональные требования

### Публичная часть (сайт турнира)

| Раздел | Описание |
|---|---|
| **Главная страница** | Баннер лиги, ближайшие матчи, последние результаты, дайджест новостей, виджет таблицы |
| **О лиге** | Описание, регламент, документы (PDF) |
| **Турниры** | Список турниров (дивизионов) с фильтрацией по сезону. Каждый турнир содержит: регулярный чемпионат, плей-офф, группы. Турниры создаются администратором лиги (см. [Администратор лиги](#администратор-лиги)) |
| **Календарь матчей** | Расписание игр, добавленных администратором лиги в рамках каждого турнира; отображение на сайте лиги для пользователей. Фильтры по турниру, команде, дате. Статусы: запланирован / идёт / завершён |

| **Турнирные таблицы** | Автоматический расчёт: очки, победы, поражения, разница шайб/мячей/голов, места |
| **Команды** | Карточка команды: логотип, состав, статистика, история матчей |
| **Игроки** | Карточка игрока: фото, амплуа, номер, личная статистика (голы, передачи, штрафы), история команд |
| **Судьи / Тренеры** | Справочник с профилями |
| **Новости** | Лента новостей с пагинацией, тегами, превью изображений |
| **Медиа** | Фотогалереи (привязка к матчам/турнирам) и видео (YouTube / VK Video embed) |
| **Заявка на участие** | Форма подачи заявки команды на турнир |

### Администратор лиги

Требования к возможностям **администратора лиги** (не суперадмина платформы). Во всех сценариях учитывается **вид спорта**, заданный для лиги (или турнира): терминология, доступные амплуа, набор полей в карточках игроков и состав протокольных событий соответствуют выбранному виду спорта.

| ID | Требование |
|---|---|
| **ФТ-AL-1** | Администратор лиги может **создавать турниры** (сезоны, дивизионы, этапы — в границах модели продукта), управлять их параметрами и жизненным циклом. |
| **ФТ-AL-2** | Администратор лиги может **добавлять игры в календарь** в рамках **каждого турнира** (дата, время, площадка, участники и пр.). Эти игры **отображаются пользователям** на публичном сайте лиги в расписании и связанных разделах. |
| **ФТ-AL-3** | Администратор лиги может **создавать команды** и **создавать игроков** с атрибутами: **ФИО**, **амплуа** (роль/позиция в соответствии с видом спорта), **возраст** (или дата рождения с вычислением возраста — по решению реализации). |
| **ФТ-AL-4** | **Вид спорта** задаётся на уровне лиги (и при необходимости уточняется для турнира) и определяет: допустимые значения амплуа, набор статистики и **типы протокольных событий** для матчей этой лиги/турнира. |
| **ФТ-AL-5** | Администратор лиги может **вносить результаты матчей** и полный **протокол** с **всеми протокольными событиями**, релевантными для выбранного вида спорта (голы, передачи, удаления, нарушения, смены, тайм-ауты и т.д. — по предустановке для вида спорта). Пересчёт таблиц и статистики выполняется после сохранения протокола. |
| **ФТ-AL-6** | При **регистрации** будущего администратора лиги после успешного ввода **проверочного кода (OTP)** выполняется автоматический вход и **немедленный переход** на экран **настроек сайта лиги** в админ-панели этой лиги (раздел конструктора), без промежуточного дашборда и без лишних шагов. См. [Регистрация и первый вход](#регистрация-и-первый-вход). |
| **ФТ-AL-7** | На экране **настроек сайта лиги** помимо **выбора темы оформления** доступны: загрузка **эмблемы лиги**; добавление **фотографий для новостей турнира** (иллюстрации к новостям в привязке к турниру — для превью и тела публикаций). |

### Регистрация и первый вход

Функциональное требование **ФТ-AL-6**: после подтверждения email по OTP пользователь оказывается сразу в контексте своей лиги на странице настройки сайта.

| Этап | Поведение |
|---|---|
| **После ввода OTP** | Статус учётной записи `ACTIVE`, создание сессии, редирект на `{slug}.sporthub.ru/admin/settings/site` (или эквивалент: `/admin/settings` с активной вкладкой «Сайт» / «Оформление») — **настройки сайта лиги**, не на `/admin/dashboard`. |

## Нефункциональные требования

### NFR-1. Производительность

| Метрика | Целевое значение | Метод измерения |
|---|---|---|
| **Time to First Byte (TTFB)** публичных страниц | ≤ 200 мс (p95) | Lighthouse CI, Vercel Analytics |
| **Largest Contentful Paint (LCP)** | ≤ 2.5 сек | Core Web Vitals |
| **First Input Delay (FID)** | ≤ 100 мс | Core Web Vitals |
| **Cumulative Layout Shift (CLS)** | ≤ 0.1 | Core Web Vitals |
| Время ответа API (CRUD операции в админке) | ≤ 500 мс (p95) | APM (Sentry, OpenTelemetry) |
| Время полнотекстового поиска | ≤ 300 мс (p95) | PostgreSQL query log |
| Пересчёт турнирной таблицы после сохранения протокола | ≤ 1 сек | Server Action timing |

**Стратегия достижения:**
- SSR + ISR для публичных страниц (кэширование на уровне Next.js, revalidate 30–60 сек)
- Redis-кэш для часто запрашиваемых данных (таблицы, ближайшие матчи)
- Индексация PostgreSQL: GIN-индексы для FTS, B-tree для FK и фильтров
- Lazy loading изображений, оптимизация через `next/image` (WebP, srcset)
- Code splitting: каждая страница — отдельный chunk

### NFR-2. Масштабируемость

| Параметр | Начальная нагрузка (MVP) | Целевая нагрузка (12 мес) |
|---|---|---|
| Количество лиг | 10–50 | 500+ |
| Команд на лигу | 5–20 | 50 |
| Игроков на лигу | 50–300 | 1 000 |
| Матчей на лигу за сезон | 30–100 | 500 |
| Одновременных пользователей (все лиги) | 50 | 1 000 |
| Объём БД | < 1 ГБ | 10–50 ГБ |

**Стратегия:**
- Горизонтальное масштабирование: несколько реплик Next.js за load balancer
- Connection pooling: PgBouncer или Prisma Accelerate
- Партиционирование таблиц `Match`, `MatchEvent`, `AuditLog` по дате при превышении 10М строк
- Вынос медиафайлов в S3-совместимое хранилище + CDN
- При достижении предела FTS (>100К записей) — миграция на Meilisearch

### NFR-3. Доступность (Availability)

| Метрика | Целевое значение |
|---|---|
| **Uptime** | ≥ 99.5% (≤ 43.8 ч простоя/год) |
| **Плановое обслуживание** | ≤ 30 мин/мес, в ночное время (UTC+3) |
| **RTO** (Recovery Time Objective) | ≤ 1 час |
| **RPO** (Recovery Point Objective) | Зависит от настроенной частоты бэкапов |

**Стратегия:**
- Docker health checks + автоматический restart
- Мониторинг: Uptime Robot / Better Stack для алертов
- Zero-downtime deploys через rolling update (Docker Swarm) или Vercel preview → promote

### NFR-4. Безопасность

| Требование | Реализация |
|---|---|
| **Аутентификация** | Auth.js v5, JWT-токены, httpOnly cookies |
| **Хеширование паролей** | bcrypt, salt rounds = 12 |
| **Хранение секретов** | Только в `.env`, никогда в коде или репозитории |
| **CSRF-защита** | Встроенная в Next.js Server Actions (origin check) |
| **XSS-защита** | React auto-escaping; sanitize HTML-контента из WYSIWYG (DOMPurify) |
| **SQL-инъекции** | Prisma ORM (parameterized queries); raw-запросы только через `$queryRaw` с параметрами |
| **Rate limiting** | На эндпоинтах: login (5 попыток/мин), OTP (3 попытки/код), API search (30 req/мин) |
| **Заголовки безопасности** | `Strict-Transport-Security`, `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy` — через `next.config.ts` headers |
| **Загрузка файлов** | Валидация MIME-type и размера на сервере; макс. 5 МБ для изображений, 50 МБ для документов |
| **Изоляция данных тенантов** | Все запросы фильтруются по `leagueId`; middleware запрещает кросс-лиговый доступ |
| **Audit log** | Все действия суперадмина и критические действия админов логируются с IP-адресом |
| **Блокировка brute force** | Счётчик `failedAttempts` на `User`; блокировка на 15 мин после 5 неудачных попыток |
| **Зависимости** | `pnpm audit` в CI; Dependabot/Renovate для автообновлений |

### NFR-7. Наблюдаемость (Observability)

| Компонент | Инструмент | Что отслеживаем |
|---|---|---|
| **Логирование** | Pino (structured JSON logs) | Ошибки, предупреждения, ключевые бизнес-события |
| **Мониторинг ошибок** | Sentry | Runtime-ошибки клиента и сервера, stack trace, breadcrumbs |
| **APM** | Sentry Performance / Vercel Analytics | TTFB, LCP, FID, CLS, API latency |
| **Uptime** | Better Stack / Uptime Robot | HTTP-проверки каждые 60 сек, алерты в Telegram |
| **Метрики БД** | pg_stat_statements | Медленные запросы (> 500 мс), deadlocks, connection count |
| **Алерты** | Telegram / email | Downtime, error rate spike (>1% за 5 мин), диск > 90%, RAM > 90% |

**Уровни логирования:**

| Уровень | Когда использовать |
|---|---|
| `error` | Ошибки, требующие вмешательства (500, failed FK constraint, SMTP down) |
| `warn` | Аномалии, не блокирующие работу (rate limit hit, brute force attempt) |
| `info` | Бизнес-события (регистрация лиги, блокировка пользователя, архивация турнира) |
| `debug` | Диагностика при разработке (SQL-запросы, cache hit/miss) — отключён в production |

### Модель данных (дополнение)

```prisma
model User {
  id             String       @id @default(cuid())
  email          String       @unique
  name           String
  passwordHash   String
  status         UserStatus   @default(PENDING)
  isSuperAdmin   Boolean      @default(false)
  failedAttempts Int          @default(0)
  lockedUntil    DateTime?
  createdAt      DateTime     @default(now())
  updatedAt      DateTime     @updatedAt

  leagues        UserLeague[] // Лиги, в которых пользователь участвует
}

model UserLeague {
  id        String     @id @default(cuid())
  user      User       @relation(fields: [userId], references: [id], onDelete: Cascade)
  userId    String
  league    League     @relation(fields: [leagueId], references: [id], onDelete: Cascade)
  leagueId  String
  role      LeagueRole @default(ADMIN)
  createdAt DateTime   @default(now())

  @@unique([userId, leagueId])
}

model VerificationCode {
  id        String   @id @default(cuid())
  email     String
  codeHash  String   // SHA-256 хеш: 6-значный OTP (регистрация) или UUID-токен (сброс пароля)
  type      CodeType @default(REGISTRATION)
  expiresAt DateTime // REGISTRATION: 10 мин, PASSWORD_RESET: 30 мин
  used      Boolean  @default(false)
  createdAt DateTime @default(now())

  @@index([email, type])
  @@index([codeHash])
}

enum UserStatus {
  PENDING
  ACTIVE
  BLOCKED
}

enum LeagueRole {
  ADMIN         // Полный доступ к лиге
  EDITOR        // Ограниченный доступ (результаты, новости, медиа)
}

enum CodeType {
  REGISTRATION      // 6-значный цифровой OTP, вводится вручную
  PASSWORD_RESET    // UUID-токен, передаётся в URL (?token=...)
}
```

> **Связь User ↔ League**: Один пользователь может быть администратором/редактором **нескольких** лиг через промежуточную таблицу `UserLeague`. Роль (`ADMIN` / `EDITOR`) определяется **в контексте конкретной лиги**, а не глобально. Суперадмин — это флаг `isSuperAdmin` на уровне `User`, он не связан ни с одной лигой.
>
> **Безопасность**: Все коды и токены хранятся в единой таблице `VerificationCode` в виде SHA-256 хешей (`codeHash`). Для регистрации генерируется 6-значный цифровой OTP (TTL 10 мин), для сброса пароля — UUID-токен, передаваемый в URL (TTL 30 мин). Пароли пользователей хешируются через bcrypt с salt rounds = 12.

### Ролевая модель

В системе три уровня доступа. **Суперадмин** — глобальный флаг на уровне `User` (`isSuperAdmin: true`), управляет платформой. **Админ** и **Редактор** — роли в контексте конкретной лиги через таблицу `UserLeague`. Один пользователь может быть админом одной лиги и редактором другой.

#### Суперадмин (`isSuperAdmin: true`)

Создаётся автоматически при первом запуске (seed) с логином и паролем из переменных окружения. Не регистрируется через форму, не имеет записей в `UserLeague`. Входит через отдельную страницу `sporthub.ru/superadmin/login`.

**Может:**

| Действие | Описание |
|---|---|
| **Просмотр всех лиг** | Список лиг с фильтрацией, поиском, статистикой |
| **Блокировка / разблокировка аккаунтов** | Изменение `status` пользователя на `BLOCKED` / `ACTIVE` |
| **Просмотр пользователей** | Список всех зарегистрированных пользователей по всем лигам |
| **Аналитика платформы** | Общая статистика: количество лиг, пользователей, матчей, динамика регистраций |
| **Модерация контента** | Просмотр новостей и медиа всех лиг, скрытие нарушающего контента (флаг `hidden`) |
| **Системные уведомления** | Рассылка email всем админам лиг (технические работы, обновления) |
| **Управление тарифами** | Настройка лимитов: макс. команд, игроков, объём хранилища на лигу |
| **Логи действий** | Просмотр audit log по всей платформе |

**Не может:**

| Ограничение | Причина |
|---|---|
| Редактировать данные лиг (команды, игроки, матчи, таблицы) | Данные принадлежат организатору лиги |
| Удалять лиги | Только блокировка аккаунта владельца (лига становится недоступна) |
| Изменять настройки конструктора лиг (тему, навигацию, разделы) | Это прерогатива админа лиги |
| Создавать турниры, вносить результаты | Суперадмин не является организатором |

#### Админ лиги (`UserLeague.role = ADMIN`)

Организатор, зарегистрировавший лигу. Полный доступ ко всем данным **этой** лиги. Один пользователь может быть админом нескольких лиг — при входе выбирает, в какую лигу перейти.

| Действие | Доступ |
|---|---|
| CRUD турниров, команд, игроков, матчей | Полный |
| Внесение результатов и статистики | Полный |
| Управление новостями и медиа | Полный |
| Настройки конструктора (тема, навигация, разделы) | Полный |
| Обработка заявок | Полный |
| Управление редакторами своей лиги | Приглашение / удаление |

#### Редактор лиги (`UserLeague.role = EDITOR`)

Приглашается админом лиги для помощи с контентом. Может быть редактором в нескольких лигах одновременно.

| Действие | Доступ |

| Управление новостями | Создание / редактирование |
| Загрузка медиа | Полный |
| CRUD команд, игроков | Только просмотр |
| Настройки конструктора | Нет доступа |
| Управление турнирами | Нет доступа |

## Мультитенантность и хостинг

Платформа использует **мультитенантную архитектуру** — все сайты пользователей обслуживаются одним инстансом Next.js-приложения. Разделение данных происходит по поддоменам (аналогично `ehl.join.hockey`, `mfl.join.hockey`).

### Принцип работы

1. Каждая лига при создании получает уникальный **slug** (например, `ehl`)
2. Сайт лиги доступен по поддомену: `ehl.sporthub.ru`
3. **Middleware** Next.js перехватывает каждый запрос, извлекает поддомен из `Host`-заголовка и определяет `leagueSlug`
4. `leagueSlug` пробрасывается в контекст приложения (через cookies / headers / серверный контекст)
5. Все запросы к БД автоматически фильтруются по `leagueId`

### Middleware (маршрутизация по поддомену)

```typescript
// src/middleware.ts
import { NextRequest, NextResponse } from "next/server";

const ROOT_DOMAIN = process.env.ROOT_DOMAIN!; // "sporthub.ru"

function extractSubdomain(hostname: string): string | null {
  // Локальная разработка: ehl.localhost:3000 → "ehl"
  const localhostMatch = hostname.match(/^(.+)\.localhost/);
  if (localhostMatch) return localhostMatch[1];

  // Продакшен: ehl.sporthub.ru → "ehl"
  if (hostname.endsWith(`.${ROOT_DOMAIN}`)) {
    return hostname.replace(`.${ROOT_DOMAIN}`, "");
  }

  // Основной домен (sporthub.ru, www.sporthub.ru, localhost:3000)
  return null;
}

export function middleware(request: NextRequest) {
  const hostname = request.headers.get("host") ?? "";
  const subdomain = extractSubdomain(hostname);

  // Основной домен (или localhost без поддомена) — лендинг + суперадмин
  if (!subdomain || subdomain === "www") {
    return NextResponse.rewrite(
      new URL(`/platform${request.nextUrl.pathname}`, request.url)
    );
  }

  // Поддомен — сайт конкретной лиги
  return NextResponse.rewrite(
    new URL(`/site/${subdomain}${request.nextUrl.pathname}`, request.url)
  );
}
```

### Изоляция данных

| Уровень | Механизм |
|---|---|
| **Маршрутизация** | Middleware определяет `leagueSlug` по поддомену |
| **Данные** | Все сущности (команды, игроки, матчи, новости) содержат `leagueId` — FK на таблицу `League` |
| **Запросы** | Prisma-запросы всегда включают `where: { leagueId }`, вынесено в обёртку `getLeagueContext()` |
| **Файлы** | Загружаемые файлы хранятся в папках, разделённых по `leagueId`: `uploads/{leagueId}/...` |
| **Кэш** | Redis-ключи содержат префикс лиги: `league:{leagueId}:standings:...` |

### DNS и домены

```
*.sporthub.ru  →  A-запись на IP сервера (wildcard)
```

- **Wildcard DNS**: одна A/CNAME-запись `*.sporthub.ru` направляет все поддомены на сервер
- **Wildcard SSL**: Let's Encrypt wildcard-сертификат через DNS challenge (`*.sporthub.ru`)
- **Кастомные домены** (опционально): лига может привязать собственный домен (`myliga.com`), который проксируется на тот же инстанс

### Prisma-схема (полная)

```prisma
// ─── Лига и сезоны ──────────────────────────────────────

model League {
  id          String    @id @default(cuid())
  name        String
  slug        String    @unique
  logo        String?
  description String?
  sportType   SportType
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  seasons     Season[]
  tournaments Tournament[]
  teams       Team[]
  players     Player[]
  coaches     Coach[]
  referees    Referee[]
  news        News[]
  media       Media[]
  applications Application[]
  members     UserLeague[]
  matches     Match[]
  siteConfig  SiteConfig?
  plan        LeaguePlan?
}

model Season {
  id        String   @id @default(cuid())
  name      String   // "2025-2026"
  startDate DateTime
  endDate   DateTime
  isCurrent Boolean  @default(false)
  league    League   @relation(fields: [leagueId], references: [id])
  leagueId  String

  tournaments Tournament[]
  rosters     TeamRoster[]

  @@unique([leagueId, name])
}

// ─── Турниры и группы ───────────────────────────────────

model Tournament {
  id         String         @id @default(cuid())
  name       String
  type       TournamentType
  archivedAt DateTime?      // null = активный, дата = архив
  league     League         @relation(fields: [leagueId], references: [id], onDelete: Restrict)
  leagueId   String
  season     Season         @relation(fields: [seasonId], references: [id], onDelete: Restrict)
  seasonId   String

  groups      Group[]
  matches     Match[]
  standings   Standing[]
  applications Application[]
}

model Group {
  id           String     @id @default(cuid())
  name         String     // "Группа А"
  archivedAt   DateTime?
  tournament   Tournament @relation(fields: [tournamentId], references: [id], onDelete: Restrict)
  tournamentId String

  matches  Match[]
  standings Standing[]
}

// ─── Матчи и события ────────────────────────────────────

model Match {
  id           String      @id @default(cuid())
  datetime     DateTime
  status       MatchStatus @default(SCHEDULED)
  archivedAt   DateTime?
  homeTeam     Team        @relation("HomeMatches", fields: [homeTeamId], references: [id], onDelete: Restrict)
  homeTeamId   String
  awayTeam     Team        @relation("AwayMatches", fields: [awayTeamId], references: [id], onDelete: Restrict)
  awayTeamId   String
  homeScore    Int?
  awayScore    Int?
  overtime     Boolean     @default(false)
  shootout     Boolean     @default(false)
  venue        String?
  round        Int?
  tournament   Tournament  @relation(fields: [tournamentId], references: [id], onDelete: Restrict)
  tournamentId String
  group        Group?      @relation(fields: [groupId], references: [id], onDelete: SetNull)
  groupId      String?
  league       League      @relation(fields: [leagueId], references: [id], onDelete: Restrict)
  leagueId     String      // денормализация: быстрая фильтрация "все матчи лиги"

  events      MatchEvent[]
  penalties   Penalty[]
  goalieStats GoalieStats[]
  media       Media[]

  @@index([leagueId, datetime])
  @@index([tournamentId])
}

model MatchEvent {
  id       String         @id @default(cuid())
  type     MatchEventType
  match    Match          @relation(fields: [matchId], references: [id], onDelete: Cascade)
  matchId  String
  player   Player         @relation(fields: [playerId], references: [id], onDelete: Restrict)
  playerId String
  team     Team           @relation(fields: [teamId], references: [id], onDelete: Restrict)
  teamId   String
  period   Int            // 1, 2, 3, OT
  time     String         // "12:34"

  @@index([matchId])
}

model Penalty {
  id       String @id @default(cuid())
  match    Match  @relation(fields: [matchId], references: [id], onDelete: Cascade)
  matchId  String
  player   Player @relation(fields: [playerId], references: [id], onDelete: Restrict)
  playerId String
  team     Team   @relation(fields: [teamId], references: [id], onDelete: Restrict)
  teamId   String
  minutes  Int
  reason   String
  period   Int
  time     String

  @@index([matchId])
}

model GoalieStats {
  id           String  @id @default(cuid())
  match        Match   @relation(fields: [matchId], references: [id], onDelete: Cascade)
  matchId      String
  player       Player  @relation(fields: [playerId], references: [id], onDelete: Restrict)
  playerId     String
  team         Team    @relation(fields: [teamId], references: [id], onDelete: Restrict)
  teamId       String
  saves        Int     @default(0)
  goalsAgainst Int     @default(0)
  shutout      Boolean @default(false)

  @@unique([matchId, playerId])
}

model Team {

  name         String
  logo         String?
  city         String?
  foundedYear  Int?
  description  String?
  contactEmail String?
  archivedAt   DateTime?
  league       League    @relation(fields: [leagueId], references: [id], onDelete: Restrict)

  homeMatches  Match[]       @relation("HomeMatches")
  awayMatches  Match[]       @relation("AwayMatches")

  goalieStats  GoalieStats[]
  standings    Standing[]

// ─── Турнирная таблица ──────────────────────────────────

model Standing {
  id           String     @id @default(cuid())
  team         Team       @relation(fields: [teamId], references: [id], onDelete: Cascade)
  teamId       String
  tournament   Tournament @relation(fields: [tournamentId], references: [id], onDelete: Cascade)
  tournamentId String
  group        Group?     @relation(fields: [groupId], references: [id], onDelete: SetNull)
  groupId      String?
  gamesPlayed  Int        @default(0)
  wins         Int        @default(0)
  losses       Int        @default(0)
```

## Архивация и защита от удаления

Физическое удаление данных запрещено для сущностей, которые имеют зависимости. Вместо этого используется **мягкое удаление** (soft delete) через поле `archivedAt`.

### Принцип

```
Удаление запрошено       →  archivedAt = NOW()
                             Все дочерние сущности также получают archivedAt = NOW()
                             На публичном сайте архивные записи не отображаются
                             В админке — отображаются с пометкой "Архив"

Восстановление           →  archivedAt = NULL
                             Дочерние сущности восстанавливаются вместе
```

### Модели с `archivedAt`

| Модель | При архивации каскадно архивируются |
|---|---|
| `Tournament` | → `Group`, `Match` (и их события), `Standing`, `Application` |
| `Team` | → `TeamRoster`, `Standing` (записи этой команды) |
| `Player` | → `TeamRoster` (записи этого игрока) |
| `Match` | → `MatchEvent`, `Penalty`, `GoalieStats` |
| `Group` | → `Match` (матчи этой группы), `Standing` |
| `Season` | → Нельзя архивировать пока есть активные турниры |

### Правила `onDelete` в Prisma

| Связь | Стратегия | Причина |
|---|---|---|
| Match → MatchEvent, Penalty, GoalieStats | `Cascade` | Дочерние данные протокола не имеют смысла без матча |
| Match → Team (home/away) | `Restrict` | Нельзя удалить команду, участвовавшую в матчах |
| Match → Tournament | `Restrict` | Нельзя удалить турнир, в котором есть матчи |
| MatchEvent/Penalty/GoalieStats → Player | `Restrict` | Нельзя удалить игрока, у которого есть статистика |
| MatchEvent/Penalty/GoalieStats → Team | `Restrict` | Нельзя удалить команду, у которой есть события |
| Group → Tournament | `Restrict` | Нельзя удалить турнир, в котором есть группы |
| Standing → Tournament | `Cascade` | Таблица пересчитывается, запись удаляется с турниром |
| Standing → Group | `SetNull` | При удалении группы строка таблицы остаётся |
| TeamRoster → Team, Player | `Cascade` | При архивации команды/игрока состав удаляется |
| Coach → Team | `SetNull` | При архивации команды тренер остаётся, но без команды |
| Media → Match | `SetNull` | При архивации матча медиа остаётся в общей галерее |
| Application → Tournament | `Restrict` | Нельзя удалить турнир с необработанными заявками |
| UserLeague → User, League | `Cascade` | При удалении пользователя или лиги связь удаляется |
| Все → League | `Restrict` | Лигу нельзя удалить, только заблокировать владельца |

### Фильтрация в запросах

Все публичные запросы автоматически добавляют `WHERE archivedAt IS NULL`:

```typescript
// src/lib/prisma-extensions.ts
import { Prisma } from "@prisma/client";

export const softDeleteExtension = Prisma.defineExtension({
  query: {
    $allModels: {
      async findMany({ args, query }) {
        args.where = { ...args.where, archivedAt: null };
        return query(args);
      },
      async findFirst({ args, query }) {
        args.where = { ...args.where, archivedAt: null };
        return query(args);
      },
    },
  },
});
```

В админке фильтр `archivedAt IS NULL` не применяется — архивные записи показываются с визуальной пометкой и кнопкой «Восстановить».

### Server Action: архивация турнира

```typescript
// src/actions/tournaments.ts
export async function archiveTournament(tournamentId: string) {
  const now = new Date();

  await prisma.$transaction([
    prisma.tournament.update({
      where: { id: tournamentId },
      data: { archivedAt: now },
    }),
    prisma.group.updateMany({
      where: { tournamentId },
      data: { archivedAt: now },
    }),
    prisma.match.updateMany({
      where: { tournamentId },
      data: { archivedAt: now },
    }),
  ]);
}
```

## Поиск

Платформа использует **PostgreSQL Full-Text Search** — встроенный полнотекстовый поиск без внешних зависимостей. Для русского и английского языков.

### Стратегия

| Уровень | Технология | Описание |
|---|---|---|
| **Быстрый фильтр** | `ILIKE` + `@@index` | Поиск по точному вхождению подстроки (автодополнение в формах) |
| **Полнотекстовый** | PostgreSQL `tsvector` + `tsquery` | Морфологический поиск с ранжированием (публичная строка поиска) |
| **Миграция на внешний сервис** | Meilisearch (опционально) | При масштабировании свыше ~100К записей |

### Где доступен поиск

**Публичный сайт** (`{slug}.sporthub.ru`):

| Место | Что ищем | Метод |
|---|---|---|
| **Глобальная строка поиска** (хедер) | Игроки, команды, новости — одним запросом | Full-text, результаты группируются по типу |
| `/players` | Игроки по ФИО | Full-text + фильтры (команда, амплуа) |
| `/teams` | Команды по названию, городу | ILIKE |
| `/news` | Новости по заголовку, содержимому, тегам | Full-text |
| `/calendar`, `/results` | Матчи по названию команды | ILIKE на join Team |

**Админ-панель** (`{slug}.sporthub.ru/admin`):

| Место | Что ищем | Метод |
|---|---|---|
| Управление игроками | ФИО, дата рождения | Full-text + фильтры |
| Управление командами | Название, город, email | ILIKE |
| Протокол матча (выбор игрока) | ФИО игрока из состава команды | ILIKE с debounce |
| Управление заявками | Название команды, контакт | ILIKE |

**Панель суперадмина** (`sporthub.ru/superadmin`):

| Место | Что ищем | Метод |
|---|---|---|
| Пользователи | Email, имя | ILIKE |
| Лиги | Название, slug | ILIKE |
| Audit log | Email, действие | ILIKE + фильтры по дате |

### Реализация: PostgreSQL Full-Text Search

Добавление `tsvector`-колонок для моделей с полнотекстовым поиском:

```sql
-- Миграция: добавление поискового индекса для игроков
ALTER TABLE "Player"
  ADD COLUMN "searchVector" tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('russian', coalesce("lastName", '')), 'A') ||
    setweight(to_tsvector('russian', coalesce("firstName", '')), 'B') ||
    setweight(to_tsvector('russian', coalesce("middleName", '')), 'C')
  ) STORED;

CREATE INDEX "Player_search_idx" ON "Player" USING GIN ("searchVector");

-- Миграция: поисковый индекс для новостей
ALTER TABLE "News"
  ADD COLUMN "searchVector" tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('russian', coalesce("title", '')), 'A') ||
    setweight(to_tsvector('russian', coalesce("content", '')), 'C')
  ) STORED;

CREATE INDEX "News_search_idx" ON "News" USING GIN ("searchVector");

-- Миграция: поисковый индекс для команд
ALTER TABLE "Team"
  ADD COLUMN "searchVector" tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('russian', coalesce("name", '')), 'A') ||
    setweight(to_tsvector('russian', coalesce("city", '')), 'B')
  ) STORED;

CREATE INDEX "Team_search_idx" ON "Team" USING GIN ("searchVector");
```

### Поисковый запрос (пример)

```typescript
// src/lib/search.ts
import { prisma } from "./prisma";

export async function searchPlayers(leagueId: string, query: string) {
  const tsQuery = query
    .trim()
    .split(/\s+/)
    .map((word) => `${word}:*`)
    .join(" & ");

  return prisma.$queryRaw`
    SELECT id, "firstName", "lastName", "middleName", photo, role
    FROM "Player"
    WHERE "leagueId" = ${leagueId}
      AND "archivedAt" IS NULL
      AND "searchVector" @@ to_tsquery('russian', ${tsQuery})
    ORDER BY ts_rank("searchVector", to_tsquery('russian', ${tsQuery})) DESC
    LIMIT 20
  `;
}
```

## Страницы и маршруты

### Особенности маршрутизации

Приложение использует **middleware rewrite** для разделения основного домена и поддоменов лиг. Все входящие запросы перенаправляются middleware на внутренние маршруты:

| Пользовательский URL | Внутренний маршрут | Описание |
|---|---|---|
| `sporthub.ru/pricing` | `/platform/pricing` | Основной домен — маркетинг и суперадмин |
| `ehl.sporthub.ru/teams` | `/site/ehl/teams` | Поддомен лиги — публичная часть |
| `ehl.sporthub.ru/admin/dashboard` | `/site/ehl/admin/dashboard` | Поддомен лиги — админ-панель |

**Ключевые правила маршрутизации:**

1. **`/superadmin`** (основной домен) — перенаправляет на `/superadmin/dashboard` (авторизованный пользователь) или `/superadmin/login` (неавторизованный)
2. **`/admin`** (основной домен) — **недоступен**, выполняется редирект на `/`. Админ-панель лиги работает **только на поддомене**: `{slug}.sporthub.ru/admin`
3. **Страницы логина** (`/superadmin/login`, `/admin/login`) рендерятся **без защитного layout** (без проверки авторизации и без сайдбара), чтобы избежать циклических редиректов

Middleware передаёт оригинальный pathname через заголовок `x-pathname`, который используется в layout-компонентах для определения текущего маршрута.

## Панель суперадмина

Расположена по скрытому маршруту `sporthub.ru/superadmin`. Суперадмин создаётся при первом запуске (seed) с логином и паролем из переменных окружения `SUPERADMIN_EMAIL` / `SUPERADMIN_PASSWORD`. Не регистрируется через публичную форму.

### Вход

- Отдельная страница `/superadmin/login` (не связана с `/admin/login` лиг)
- Проверка `isSuperAdmin: true` в middleware — если флаг не установлен, возвращается 404 (а не 403, чтобы не раскрывать существование маршрута)

### Дашборд платформы

- Общее количество лиг, пользователей, матчей
- График регистраций новых лиг по дням/неделям
- Топ-5 лиг по активности (матчи за последний месяц)
- Последние зарегистрированные лиги
- Заблокированные аккаунты

### Управление лигами

- Таблица всех лиг: название, slug, вид спорта, дата создания, статус владельца, количество команд/матчей
- Фильтры: по виду спорта, дате создания, статусу
- Переход на публичный сайт лиги (ссылка `{slug}.sporthub.ru`)
- Суперадмин **не может** редактировать данные лиги (команды, игроки, турниры)

### Управление пользователями

- Таблица: email, имя, роль, лига, статус, дата регистрации, последний вход
- **Блокировка**: кнопка → статус `BLOCKED` → пользователь не может войти, сайт лиги показывает заглушку «Лига временно недоступна»
- **Разблокировка**: обратное действие
- Поиск по email / имени
- Фильтр по статусу (`ACTIVE` / `BLOCKED` / `PENDING`)

### Модерация контента

- Лента новостей всех лиг с превью
- Галерея медиа всех лиг
- Кнопка «Скрыть» — устанавливает флаг `hidden: true`, контент перестаёт отображаться на публичном сайте
- Причина скрытия (текстовое поле) → отправляется email-уведомление админу лиги

### Системные уведомления

- Форма рассылки email всем админам лиг (или выборочно)
- Шаблоны: технические работы, обновление платформы, изменение условий
- История отправленных уведомлений

### Управление тарифами

- Настройка лимитов по умолчанию и для конкретных лиг:
  - Максимальное количество команд
  - Максимальное количество игроков
  - Объём хранилища для медиа (МБ)
  - Количество турниров за сезон

### Audit Log

- Журнал всех действий на платформе: кто, что, когда, в какой лиге
- Фильтры по типу действия, пользователю, лиге, дате
- Типы событий: `USER_BLOCKED`, `USER_UNBLOCKED`, `LEAGUE_CREATED`, `CONTENT_HIDDEN`, `NOTIFICATION_SENT` и др.

## Админ-панель лиги

Расположена по скрытому маршруту `{slug}.sporthub.ru/admin`. Доступна только ролям `ADMIN` и `EDITOR` (с разными правами, см. [матрицу прав](#матрица-прав-доступа)).

### Защита доступа

- Маршрут `/admin` не отображается в навигации публичного сайта
- Доступ только по прямой ссылке
- Аутентификация через Auth.js (Credentials Provider)
- Middleware проверяет: (1) наличие сессии, (2) наличие записи `UserLeague` для текущей лиги, (3) роль в этой записи (`ADMIN` или `EDITOR`)
- `EDITOR` видит только разрешённые разделы (результаты, новости, медиа)

### Ключевые возможности

**Дашборд** — сводная статистика: количество команд, игроков, матчей; ближайшие матчи; последние заявки.

**Протокол матча** — основная форма ввода данных:
1. Выбор матча из расписания
2. Ввод счёта по периодам
3. Добавление событий: голы (кто забил, кто ассистировал, время, период)
4. Штрафы (игрок, минуты, причина)
5. Вратарская статистика (сэйвы, пропущенные)
6. Автоматический пересчёт таблицы после сохранения

**Конструктор сайта** (только `ADMIN`) — визуальная настройка:
1. Выбор темы оформления из шаблонов
2. Настройка цветовой палитры (primary, secondary, accent)
3. Drag & drop блоков на главной странице
4. Конфигурация навигации (пункты меню, порядок, вложенность)
5. Превью изменений в реальном времени

**Управление редакторами** (только `ADMIN`):
1. Приглашение редактора по email (отправляется ссылка-приглашение)
2. Список редакторов лиги
3. Удаление редактора (отзыв доступа)

## Установка и запуск

### Предварительные требования

- Node.js >= 20.x
- pnpm >= 9.x (рекомендуемый пакетный менеджер)
- Docker и Docker Compose (для БД)

### Локальная разработка

```bash
# Клонирование репозитория
git clone https://github.com/your-org/sporthub.git
cd sporthub

# Установка зависимостей
pnpm install

# Запуск PostgreSQL и Redis через Docker
docker compose up -d

# Копирование переменных окружения
cp .env.example .env
# Заполнить .env (см. раздел ниже)

# Применение миграций и генерация Prisma Client
pnpm prisma migrate dev
pnpm prisma generate

# Сидинг начальных данных (создаёт суперадмина из .env)
pnpm prisma db seed

# Запуск dev-сервера
pnpm dev
```

Приложение будет доступно по адресу `http://localhost:3000`.

**Доступ к панелям управления:**

| Панель | URL | Описание |
|---|---|---|
| Суперадмин | `http://localhost:3000/superadmin/login` | Вход суперадмина (учётные данные из `.env`) |
| Суперадмин (дашборд) | `http://localhost:3000/superadmin/dashboard` | Требуется авторизация |
| Админка лиги | `http://{slug}.localhost:3000/admin/login` | Вход админа лиги (через поддомен) |
| Админка лиги (дашборд) | `http://{slug}.localhost:3000/admin/dashboard` | Требуется авторизация |

> **Важно:** Маршрут `/admin` работает **только на поддомене** лиги. На основном домене (`localhost:3000/admin`) будет выполнен редирект на главную страницу.

Seed-скрипт автоматически создаёт суперадмина:

```typescript
// prisma/seed.ts (фрагмент)
import { hash } from "bcrypt";
import { prisma } from "../src/lib/prisma";

async function main() {
  const existing = await prisma.user.findFirst({
    where: { isSuperAdmin: true },
  });

  if (!existing) {
    await prisma.user.create({
      data: {
        email: process.env.SUPERADMIN_EMAIL!,
        name: "Super Admin",
        passwordHash: await hash(process.env.SUPERADMIN_PASSWORD!, 12),
        isSuperAdmin: true,
        status: "ACTIVE",
      },
    });
    console.log("Super admin created");
  }
}
```

## Деплой

### Docker (рекомендуемый для VPS)

```bash
docker compose -f docker-compose.prod.yml up -d
```

`docker-compose.prod.yml` включает:
- Next.js приложение (multi-stage build) — один контейнер обслуживает все лиги
- PostgreSQL 16 с persistent volume
- Redis
- Nginx в качестве reverse proxy с wildcard SSL

**Настройка DNS и SSL для мультитенантности:**

```nginx
# /etc/nginx/sites-available/sporthub
server {
    listen 443 ssl;
    server_name sporthub.ru *.sporthub.ru;

    ssl_certificate     /etc/letsencrypt/live/sporthub.ru/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/sporthub.ru/privkey.pem;

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

```bash
# Wildcard SSL через certbot + DNS challenge
certbot certonly --manual --preferred-challenges dns \
  -d "sporthub.ru" -d "*.sporthub.ru"
```

```
# DNS (у регистратора домена)
*.sporthub.ru.   A   <IP сервера>
sporthub.ru.     A   <IP сервера>
```

### Vercel

```bash
# Установить Vercel CLI
pnpm i -g vercel

# Деплой
vercel --prod
```

При деплое на Vercel:
- **Neon** или **Supabase** для PostgreSQL

- Wildcard-домен `*.sporthub.ru` настраивается в Vercel Dashboard → Domains
